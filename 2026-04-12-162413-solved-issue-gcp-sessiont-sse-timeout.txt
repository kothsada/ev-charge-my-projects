  ---
  Root Causes

  You have 3 layered problems that combine to kill SSE streams on GCP:

  1. Node.js keepAliveTimeout is too short (primary cause)

  GCP's L7 Load Balancer keeps backend TCP connections alive for 600 seconds and
   reuses them. Node.js defaults keepAliveTimeout to 5 seconds. When Node.js
  closes the connection, GCP LB tries to send a new request on a dead socket ‚Üí
  RST packet ‚Üí GCP returns 502/broken pipe which kills the SSE stream. main.ts
  never configures this.

  2. X-Accel-Buffering: no does nothing on GCP

  That header is nginx-specific. The GCE Ingress uses Google's own Envoy-based
  L7 LB ‚ÄĒ it ignores that header entirely. The correct way to prevent GCP LB
  from buffering a streaming response is Cache-Control: no-cache, no-transform
  (already partially there via NestJS @Sse(), but no-transform is missing).

  3. Node.js has no headersTimeout guard

  Without headersTimeout > keepAliveTimeout, there's a race window where GCP
  sends headers on a connection Node.js is about to close.

  Fix 1 ‚ÄĒ main.ts: configure HTTP server timeouts

  Question 1: How does the system know when charging finishes / battery is full?

  You don't detect it in the app ‚ÄĒ the charger hardware tells you via OCPP.

  EV Battery Full
        ‚Ēā
        ‚Ėľ
  Charger stops current flow internally
        ‚Ēā
        ‚Ėľ  (OCPP 1.6)
  StopTransaction { stopReason: "EVDisconnected" | "Local" | "Other" }
        ‚Ēā        (charger reports its reason ‚ÄĒ "Full" is NOT a standard OCPP 1.6
   reason)
        ‚Ėľ
  OCPP Service (panda-ev-ocpp)
    ‚Üí publishes transaction.stopped to PANDA_EV_QUEUE
        ‚Ēā
        ‚Ėľ
  Mobile API OcppConsumerService
    ‚Üí marks session COMPLETED, deducts wallet
    ‚Üí Redis publish to session:ended:{sessionId}
        ‚Ēā
        ‚Ėľ
  SSE stream sends: { ended: true, status: "COMPLETED", sessionId }
        ‚Ēā
        ‚Ėľ
  Client closes EventSource

  The battery "full" case specifically: OCPP 1.6 has no stopReason: "Full". What
   actually happens:
  - The EV tells the charger via the physical charging protocol (CCS2, GBT,
  CHAdeMO) that it's full
  - The charger stops the session and sends StopTransaction with reason usually
  "Local" or "Other" (charger-dependent behavior)
  - Then sends StatusNotification with connector status Finishing ‚Üí Available

  You cannot reliably distinguish "battery full" from "user manually stopped" at
   the OCPP 1.6 level. Some charger brands encode it differently. The stopReason
   in your transaction.stopped event carries whatever the charger reports.

  For the mobile app, the correct UX is: session COMPLETED = charging is done, 
  reason doesn't matter. You push an FCM notification at that point.

  ---
  Question 2: Why is keepAliveTimeout = 620s enough for any charging duration?

  keepAliveTimeout and the SSE stream duration are completely separate things:

  keepAliveTimeout = how long Node.js keeps an IDLE connection open
                     (no active request happening)

  SSE stream        = an ACTIVE response that is streaming data
                      keepAliveTimeout does NOT apply here

  Here's the timeline visually:

  Client connects ‚Üí SSE stream starts (ACTIVE response)
  ‚Ēā
  ‚Ēā   ‚Üź heartbeat every 15s          ‚Üź keepAliveTimeout irrelevant
  ‚Ēā   ‚Üź meter values from charger       because connection is ACTIVE
  ‚Ēā   ‚Üź heartbeat ...
  ‚Ēā
  Session ends ‚Üí SSE sends { ended: true } ‚Üí response completes
  ‚Ēā
  ‚Ēā   ‚Üź NOW keepAliveTimeout starts    ‚Üź GCP keeps TCP alive 600s
  ‚Ēā   ‚Üź 620s window: Node.js keeps       Node.js keeps it 620s
  ‚Ēā      the socket open too             They agree, no RST
  ‚Ēā
  Next API call (e.g. /live poll, GET /charging-sessions)
  uses the same TCP connection

  So for charging session duration:
  - BackendConfig.timeoutSec: 3600 (1 hour) = what limits the SSE stream itself
  - keepAliveTimeout: 620s = only protects the idle socket after the stream ends

  If a session lasts more than 1 hour, the SSE stream will be cut by GCP LB
  regardless. But that is fine by design ‚ÄĒ the mobile app active charging screen
   is for when the user is watching. For overnight or long sessions:
